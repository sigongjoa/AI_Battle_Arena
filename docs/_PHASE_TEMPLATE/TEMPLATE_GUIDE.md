## 📋 개요

이 디렉토리는 AI Battle Arena 프로젝트의 모든 Phase에서 사용할 표준 문서 템플릿을 제공합니다.

**목적**:
- Phase 진행 과정의 일관성 유지
- 필요한 정보의 누락 방지
- 팀 간 의사소통 효율화
- 이후 참고자료로 활용

## 🔄 Phase 라이프사이클별 문서 작성 순서

### Phase 시작 단계 (Week 1)

1. **README.md 작성** (즉시)
   - Phase의 목표 정의
   - 전체 개요 작성
   - 관련 Phase 링크

2. **01_planning.md 작성** (3-5일)
   - 문제 정의 및 요구사항 분석
   - 범위 확정
   - 성공 기준 정의

3. **02_technical_spec.md 작성** (5-7일)
   - 기술 스택 선정
   - 아키텍처 설계
   - API/DB 설계

### Phase 진행 단계 (Week 2+)

4. **03_implementation_plan.md 작성** (1-2일)
   - 구현 단계 정의
   - 일정 계획
   - 역할 할당

5. **04_test_plan.md 작성** (2-3일)
   - 테스트 전략 수립
   - 테스트 케이스 정의
   - 성공 기준 확정

6. **05_progress_report.md 작성 및 갱신** (주간 - 매주 금요일)
   - 주간 진행률 업데이트
   - 블로커 및 이슈 기록
   - 위험 평가

### Phase 완료 단계

7. **06_lessons_learned.md 작성** (완료 후 1주)
   - 최종 결과 정리
   - 배운 점 문서화
   - 다음 Phase 권장사항

## 📝 작성 가이드라인

### 일반 원칙

1. **명확성**: 전문 용어 설명, 예시 포함
2. **완성도**: 적어도 60% 이상 작성 후 검토 받기
3. **정기 갱신**: 진행 상황 변화 시 즉시 반영
4. **협업**: 관련자 리뷰 및 승인 필수

### 템플릿 수정 규칙

**금지 사항**:
- ❌ 템플릿의 주요 섹션 제거
- ❌ 상태 표시 형식 변경 (🔵🟡🟢🔴)
- ❌ 테이블 구조 변경

**허용 사항**:
- ✅ Phase별 특성에 맞게 섹션 추가
- ✅ 예시 및 설명 추가
- ✅ 링크 및 참고자료 추가

### 예시: Phase 맞춤형 수정

**Phase X가 프론트엔드 담당인 경우**:
```markdown
# 02_technical_spec.md에 추가
## 🎨 UI/UX 설계

- ... (추가)

### 컴포넌트 라이브러리
- Material-UI v5
- Tailwind CSS
```

## ✍️ 문서별 작성 팁

### README.md

**작성 시간**: 30분~1시간
**담당자**: Phase 리드

**팁**:
- 경영진도 이해할 수 있는 수준으로 작성
- 시각적 요소 활용 (다이어그램, 차트)
- 다음 Phase와의 연관성 명시

**체크리스트**:
- [ ] Phase 목표 명확한가?
- [ ] 산출물이 구체적인가?
- [ ] 관련 문서 링크가 있는가?

### 01_planning.md

**작성 시간**: 1~2일
**담당자**: Phase 리드 + 기획자

**팁**:
- 요구사항은 SMART 기준으로 작성 (Specific, Measurable, Achievable, Relevant, Time-bound)
- 리스크는 확률과 영향도로 우선순위 정하기
- 의존성 관계도를 그려보기

**체크리스트**:
- [ ] 요구사항이 모두 정량화되어 있는가?
- [ ] 리스크 대응 방안이 구체적인가?
- [ ] 성공 기준이 측정 가능한가?

### 02_technical_spec.md

**작성 시간**: 2~3일
**담당자**: 아키텍트 + 기술 리드

**팁**:
- 기술 선택의 근거를 명시 (비교표 활용)
- 아키텍처 다이어그램은 필수 (ASCII art 또는 외부 링크)
- 마이그레이션 경로 미리 고려하기

**체크리스트**:
- [ ] 왜 이 기술을 선택했는가 명확한가?
- [ ] 성능 목표가 기술로 달성 가능한가?
- [ ] 확장성이 고려되어 있는가?

### 03_implementation_plan.md

**작성 시간**: 1~2일
**담당자**: 기술 리드 + 개발자

**팁**:
- 작업을 최소 1일 단위로 분해하기
- 병렬 처리 가능한 작업 식별하기
- 버퍼 시간 포함 (총 시간의 15-20%)

**체크리스트**:
- [ ] 모든 작업이 할당되어 있는가?
- [ ] 의존성이 명확한가?
- [ ] 각 작업의 예상 소요 시간이 현실적인가?

### 04_test_plan.md

**작성 시간**: 2~3일
**담당자**: QA + 개발 리드

**팁**:
- 테스트 케이스는 요구사항과 1:1 매핑되어야 함
- 성공 기준은 객관적으로 측정 가능해야 함
- 테스트 환경 준비 일정 미리 계획하기

**체크리스트**:
- [ ] 모든 요구사항에 대한 테스트 케이스가 있는가?
- [ ] 성공 기준이 자동으로 측정 가능한가?
- [ ] 테스트 환경 준비 일정이 현실적인가?

### 05_progress_report.md

**작성 시간**: 30분~1시간 (매주)
**담당자**: Phase 리드

**팁**:
- 매주 금요일 오후 4시에 작성 (1주 마무리)
- 진행률은 객관적인 지표 기반 (완료한 작업/전체 작업)
- 블로커는 즉시 경영진에 보고

**체크리스트**:
- [ ] 진행률이 정량적으로 표시되어 있는가?
- [ ] 블로커에 대한 해결책이 있는가?
- [ ] 다음 주 계획이 명확한가?

### 06_lessons_learned.md

**작성 시간**: 2~3시간 (완료 후)
**담당자**: Phase 전체 팀

**팁**:
- 팀 회고를 통해 함께 작성하기
- 긍정/부정을 균형있게 다루기
- 구체적인 개선 사항 제시하기

**체크리스트**:
- [ ] 좋았던 점과 개선할 점이 구체적인가?
- [ ] 다음 Phase를 위한 교훈이 있는가?
- [ ] 측정 가능한 개선 방안이 제시되어 있는가?

### 07_definition_of_done.md

**작성 시간**: 2~3시간 (초기)
**담당자**: Phase 리드 + QA 리드

**팁**:
- Phase 초기에 팀과 함께 DoD 정의
- 모든 작업 항목에 동일하게 적용
- 주간마다 DoD 충족도 추적
- 필요시 DoD 갱신 (변경 관리 프로세스 거쳐야 함)

**체크리스트**:
- [ ] 모든 작업 항목에 적용 가능한 기준인가?
- [ ] 측정 가능한가?
- [ ] 팀 전체가 동의했는가?
- [ ] 엄격성 수준(Level 1/2/3)이 명확한가?

### 08_test_documentation.md

**작성 시간**: 지속 갱신 (매 테스트 실행 후)
**담당자**: QA 엔지니어 + 개발자

**팁**:
- 테스트 실행 직후 결과를 즉시 문서화
- 모든 테스트 출력(pytest, npm, curl, Playwright) 포함
- TypeScript 타입으로 테스트 결과 구조화
- Playwright 스크린샷 자동 캡처

**체크리스트**:
- [ ] pytest 결과 커버리지 80% 이상인가?
- [ ] 모든 API 테스트 포함되어 있는가?
- [ ] npm test 결과가 문서화되어 있는가?
- [ ] Playwright 스크린샷이 첨부되어 있는가?
- [ ] Use Case 테스트 결과가 명확한가?

### 09_ui_ux_change_documentation.md

**작성 시간**: 지속 갱신 (모든 UI/UX 변경 후)
**담당자**: 프론트엔드 개발자 + 디자이너

**팁**:
- **중요**: 모든 UI/UX 변경마다 이 문서 업데이트 필수
- Before/After 스크린샷 필수 포함 (Playwright 자동 캡처)
- E2E 테스트 작성 및 통과 확인
- 접근성(WCAG 2.1 AA) 검증 필수
- 반응형 테스트(Desktop/Tablet/Mobile) 완료

**체크리스트**:
- [ ] Before/After 스크린샷이 있는가?
- [ ] E2E 테스트가 작성되고 통과했는가?
- [ ] 회귀 테스트를 실행했는가?
- [ ] 접근성을 검증했는가? (WCAG 2.1 AA)
- [ ] 모바일 반응형을 테스트했는가?
- [ ] QA 검수를 받았는가?
- [ ] 성능 영향을 측정했는가?

## 📌 문서 관리 규칙

### 버전 관리

모든 중요 문서에 버전 정보 추가:
```markdown
**버전**: 1.0
**최종 수정**: 2025-11-20
**상태**: 🟢 완료 / 🟡 진행중 / 🔵 계획중
```

### 문서 갱신 주기

| 문서 | 갱신 주기 | 담당자 |
|------|---------|--------|
| README | 초기 1회 | Phase 리드 |
| planning | 초기 1회 | 기획자 |
| technical_spec | 초기 1회 | 아키텍트 |
| implementation | 초기 1회 + 필요시 | 리드 개발자 |
| test_plan | 초기 1회 | QA |
| progress_report | 매주 | Phase 리드 |
| lessons_learned | 완료 후 1회 | 전체 팀 |

### 문서 리뷰 프로세스

```
작성 → 자체 검토 → 전문가 리뷰 → 경영진 검토 → 승인 → 공개
```

**각 단계 목표**:
1. **자체 검토**: 완성도 60% 이상
2. **전문가 리뷰**: 기술 정확성 검증
3. **경영진 검토**: 비즈니스 관점 검증
4. **승인**: 공식 승인 및 서명
